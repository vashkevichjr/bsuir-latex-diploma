Этот документ фиксирует предметную область и содержательные ограничения диплома.
Структура пояснительной записки берется из `docs/diploma-generation-instructions.md`
и `docs/diploma-generation-plan.md`. Тема, терминология, роли, сущности, API,
метрики и содержание разделов берутся из этого документа.

## Тема

`Программное средство реализации сервиса "Сквозная аналитика для e-commerce"`.

Проект строго образовательный и некоммерческий. Его нельзя описывать как
реальный коммерческий стартап с фактическими продажами. В экономическом разделе
допускается условная оценка экономической эффективности внедрения для
организации, но сам проект в контексте разработки является учебным.

## Назначение системы

Разрабатываемое программное средство автоматизирует процесс подключения и
сопровождения сквозной аналитики e-commerce проекта. Система предназначена для
digital/SEO-агентства или внутреннего аналитического отдела, который работает с
интернет-магазинами и помогает понять, какие источники трафика реально приводят
к событиям, покупкам, выручке и конверсии.

Правильная формулировка:

> Программное средство предназначено для автоматизации деятельности
> SEO-специалиста при подключении и сопровождении сквозной аналитики
> e-commerce проектов. Система обеспечивает регистрацию заявок клиентов,
> ведение проектов, ввод бизнес-данных в табличном виде, генерацию
> демонстрационного массива данных, проведение диагностики сайта, настройку
> источников трафика, расчет аналитических показателей, формирование
> рекомендаций и подготовку отчетов.

## Основные возможности

Система позволяет:

1. Клиенту оставить заявку на подключение сквозной аналитики.
2. Клиенту выбрать SEO-специалиста из списка активных специалистов.
3. Клиенту заполнить данные о бизнесе: сайт, ниша, регион, тип товаров,
   средний чек, цели аналитики, описание проблемы.
4. Клиенту передать данные о продажах и событиях в табличном виде: переходы,
   события, покупки, возвраты, источники трафика.
5. SEO-специалисту принять заявку в работу.
6. SEO-специалисту провести диагностику e-commerce проекта.
7. SEO-специалисту настроить модель аналитики: источники трафика, целевые
   события, этапы воронки, правила сопоставления переходов и покупок.
8. SEO-специалисту проверить качество данных клиента.
9. Системе рассчитать KPI: переходы, заказы, выручка, конверсия, средний чек,
   возвраты, эффективность источников.
10. SEO-специалисту проанализировать эффективность источников и воронку продаж.
11. SEO-специалисту сформировать рекомендации и отчет.
12. Клиенту просмотреть dashboard, отчет и рекомендации.
13. Администратору управлять пользователями, ролями, проектами, справочниками,
   резервным копированием и тестовыми данными.
14. Администратору сгенерировать массив тестовых данных для проекта, чтобы
   демонстрировать систему без внешнего e-commerce API.

## Архитектурное решение по данным

Внешнее e-commerce приложение как отдельный актор не используется. Причина:
в дипломе нужно показать работоспособное программное средство, а разработка
отдельного интернет-магазина или стороннего API выходит за рамки проекта.

В системе есть два механизма поступления данных:

1. Клиент вручную или через табличную форму вносит события, переходы, покупки
   и возвраты.
2. Администратор может сгенерировать массив тестовых данных для выбранного
   проекта.

Это позволяет демонстрировать полный цикл сквозной аналитики без разработки
внешнего интернет-магазина.

## Бизнес-ценность

Система решает проблему разрозненности данных в e-commerce аналитике. Вручную
SEO-специалисту приходится собирать данные из таблиц, рекламных кабинетов, CRM,
заказов и отчетов. Это долго, ошибочно и плохо масштабируется.

Сервис централизует:

- заявки клиентов;
- данные о бизнесе;
- источники трафика;
- события пользователей;
- покупки;
- возвраты;
- KPI;
- воронку продаж;
- отчеты;
- рекомендации.

Ценность:

- сокращение времени подготовки аналитики;
- снижение ручного труда SEO-специалиста;
- повышение прозрачности эффективности каналов;
- возможность быстро понять, какой источник трафика приносит заказы и выручку;
- формирование отчетности для клиента;
- контроль статусов заявок и проектов;
- хранение истории аналитических работ.

## Роли пользователей

В системе три основные роли.

| Роль | Описание |
|---|---|
| Клиент | Представитель e-commerce проекта. Оставляет заявку, выбирает SEO-специалиста, заполняет данные о бизнесе, передает данные о событиях и продажах, просматривает dashboard и отчет. |
| SEO-специалист | Основной рабочий пользователь системы. Принимает заявки, проводит диагностику проекта, настраивает модель аналитики, проверяет данные, анализирует KPI, формирует рекомендации и отчеты. |
| Администратор | Системный пользователь. Управляет пользователями и ролями, справочниками, проектами, создает проекты вручную, генерирует тестовые данные, выполняет резервное копирование и смотрит журнал действий. |

Не добавлять роль `Внешнее e-commerce приложение`.

## Use Cases

Нужно использовать 12 основных вариантов использования.

| UC | Название | Роль | Суть |
|---|---|---|---|
| UC-1 | Подать заявку на подключение сквозной аналитики | Клиент | Клиент создает заявку, указывает сайт, нишу, проблему, цель аналитики и отправляет ее в систему. |
| UC-2 | Выбрать SEO-специалиста | Клиент | Клиент выбирает SEO-специалиста из списка активных специалистов системы. |
| UC-3 | Заполнить данные о бизнесе | Клиент | Клиент заполняет бриф: регион, тип товаров, каналы продаж, средний чек, период анализа, основные цели. |
| UC-4 | Передать данные о продажах и событиях | Клиент | Клиент вносит данные в табличном виде: переходы, события, покупки, возвраты, источники трафика. |

| UC-6 | Принять заявку в работу | SEO-специалист | SEO-специалист открывает назначенную заявку, проверяет первичные данные и меняет статус на `В обработке`. |

Связи include:

- UC-1 включает UC-2.
- UC-1 включает UC-3.
- UC-9 включает UC-8.

## Стек технологий

Клиентская часть:

- Next.js 15;
- React 19;
- TypeScript;
- Tailwind CSS;
- Recharts для графиков;
- React Hook Form для форм;
- Zod для валидации;
- TanStack Query или встроенные механизмы Next.js для запросов к API.

Серверная часть:

- Go 1.23;
- chi для маршрутизации HTTP-запросов;
- pgx для работы с PostgreSQL;
- sqlc для генерации типобезопасного слоя доступа к данным;
- JWT-аутентификация;
- bcrypt или argon2 для хеширования паролей;
- PostgreSQL 16 как основная БД.

Инфраструктура:

- Docker Compose;
- PostgreSQL 16;
- Caddy для раздачи фронтенда, HTTPS и reverse proxy.

API:

- REST;
- JSON;
- OpenAPI-спецификация для документирования REST API.

В задании может быть требование передачи данных в JSON/XML. В проекте основной
формат - JSON. XML можно упоминать как потенциально поддерживаемый формат
представления, но основной API описывать через JSON.

Ключевая формулировка стека:

> Клиентская часть реализуется на базе Next.js 15, React 19 и TypeScript.
> Серверная часть реализуется на языке Go 1.23 с использованием маршрутизатора
> chi, драйвера pgx и генератора sqlc для типобезопасной работы с PostgreSQL 16.
> Развертывание выполняется через Docker Compose, а Caddy используется для
> раздачи web-интерфейса, HTTPS и reverse proxy.

## Стиль фронтенда

Фронтенд - современная B2B SaaS-платформа для аналитики e-commerce. Это не
лендинг и не маркетинговый сайт, а рабочая веб-система: dashboard, таблицы,
фильтры, KPI-карточки, формы заявок, графики, отчеты, статусы, боковое меню.

Визуальное направление:

- светлая тема;
- чистый белый/серый фон;
- синий основной акцент;
- карточки с мягкой тенью;
- деловой технологичный стиль;

## Модель данных

Основные сущности:

- `users`: `id`, `full_name`, `email`, `password_hash`, `role`, `is_active`,
  `created_at`.
- `applications`: `id`, `client_id`, `project_id`, `seo_specialist_id`,
  `description`, `goal`, `status`, `created_at`.
- `projects`: `id`, `client_id`, `seo_specialist_id`, `name`, `site_url`,
  `niche`, `status`, `created_at`.
- `traffic_sources`: `id`, `project_id`, `name`, `source_type`, `is_active`.
- `events`: `id`, `project_id`, `traffic_source_id`, `event_name`,
  `occurred_at`, `anonymous_id`.
- `orders`: `id`, `project_id`, `traffic_source_id`, `amount`, `status`,
  `ordered_at`.
- `refunds`: `id`, `order_id`, `amount`, `refunded_at`, `reason`.
- `reports`: `id`, `project_id`, `author_id`, `period_start`, `period_end`,
  `status`, `created_at`.

Дополнительные сущности:

- `business_profiles`;
- `recommendations`;
- `analytics_metrics`;
- `audit_logs`;
- `backups`.

Основные связи:

- пользователь с ролью client имеет много applications;
- пользователь с ролью seo_specialist имеет много applications и projects;
- project принадлежит client и seo_specialist;
- application может быть основанием для project;
- project имеет много traffic_sources, events, orders, reports,
  recommendations;
- order может иметь refunds;
- report создается через `author_id`, обычно SEO-специалист.

## AS-IS и TO-BE

AS-IS: ручной процесс подключения сквозной аналитики:

1. Клиент обращается к SEO-специалисту.
2. Клиент описывает проблему сайта.
3. SEO-специалист вручную собирает данные о сайте.
4. Клиент присылает таблицы с переходами, заказами, возвратами.
5. SEO-специалист вручную сверяет данные.
6. SEO-специалист вручную считает KPI.
7. SEO-специалист готовит отчет в таблицах/документах.
8. Клиент ожидает результата.

Проблемы AS-IS: много ручных операций, разрозненные таблицы, высокий риск
ошибок, долгая подготовка отчетов, отсутствие единого dashboard, сложность
сопоставления источников трафика и покупок.

TO-BE:

1. Клиент оставляет заявку в системе.
2. Клиент выбирает SEO-специалиста.
3. Клиент заполняет бриф e-commerce проекта.
4. SEO-специалист принимает заявку.
5. Система создает проект.
6. Клиент вносит события, переходы, покупки и возвраты.
7. Система валидирует данные.
8. SEO-специалист настраивает модель аналитики.
9. Система сопоставляет переходы с покупками.
10. Система рассчитывает KPI.
11. SEO-специалист анализирует источники и воронку.
12. SEO-специалист формирует рекомендации.
13. Система формирует отчет.
14. Клиент просматривает dashboard и отчет.

## Метрики

Система должна считать:

- переходы;
- события: просмотр товара, добавление в корзину, оформление заказа, покупка;
- заказы;
- выручку;
- конверсию: `заказы / переходы * 100%`;
- средний чек: `выручка / количество заказов`;
- возвраты;
- эффективность источников: `organic`, `paid_search`, `social`, `direct`,
  `referral`, `email`;
- ROMI: `(выручка - расходы) / расходы * 100%`;
- CPA: `расходы / количество заказов`;
- воронку: переход -> просмотр товара -> корзина -> заказ -> покупка.

Примеры проектов:

- `mebel-market.by`;
- `kitchen-plus.by`;
- `home-style.by`;
- `office-pro.by`;
- `dom-svet.by`;
- `comfort-matras.by`.

Пример KPI:

- переходы: 12 450;
- заказы: 382;
- выручка: 48 300 BYN;
- конверсия: 3,07%;
- возвраты: 2 100 BYN;
- средний чек: 126,44 BYN.

Примеры рекомендаций:

- оптимизировать страницы категорий;
- расширить семантику по высокочастотным запросам;
- повысить эффективность email-рассылок;
- снизить бюджет канала paid_social из-за низкой конверсии;
- упростить форму оформления заказа;
- улучшить внутреннюю перелинковку.
